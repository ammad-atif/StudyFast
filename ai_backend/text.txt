Composite Design Pattern
The Composite Design Pattern is a structural design pattern that lets you compose objects into tree structures to represent part-whole hierarchies. It allows clients to treat individual objects and compositions of objects uniformly.
Core Concept
Think of it like a file system: a folder can contain files OR other folders. When you want to calculate the total size, you don't care whether you're dealing with a single file or a folder containing hundreds of nested folders and files—you just call getSize() on the root, and it handles the complexity.
Key Components
Component: The base interface/abstract class that defines common operations for both leaf and composite objects
Leaf: Represents end objects that have no children (like a file)
Composite: Contains children (can be leaves or other composites) and implements operations by delegating to children (like a folder)
Real-World Examples
1. Organization Hierarchy
A company has departments
Each department can have employees (leaves) and sub-departments (composites)
Calculating total salary works the same whether you ask a department or an individual employee
2. Graphic Drawing Systems
Simple shapes (circle, rectangle) are leaves
Groups of shapes are composites
Drawing a group automatically draws all contained shapes
3. UI Component Trees
A button is a leaf
A panel containing buttons and other panels is a composite
Rendering, resizing, or hiding works uniformly
Java Implementation
java
// Component
interface Employee {
    void showDetails();
    int getSalary();
}
// Leaf
class Developer implements Employee {
    private String name;
    private int salary;
    private String position;
    public Developer(String name, int salary, String position) {
        this.name = name;
        this.salary = salary;
        this.position = position;
    }
    @Override
    public void showDetails() {
        System.out.println(position + ": " + name + " ($" + salary + ")");
    }
    @Override
    public int getSalary() {
        return salary;
    }
}
// Composite
class Department implements Employee {
    private String name;
    private List<Employee> employees = new ArrayList<>();
    public Department(String name) {
        this.name = name;
    }
    public void addEmployee(Employee employee) {
        employees.add(employee);
    }
    public void removeEmployee(Employee employee) {
        employees.remove(employee);
    }
    @Override
    public void showDetails() {
        System.out.println("\n" + name + " Department:");
        System.out.println("------------------------");
        for (Employee employee : employees) {
            employee.showDetails();
        }
        System.out.println("Total Department Salary: $" + getSalary());
    }
    @Override
    public int getSalary() {
        int totalSalary = 0;
        for (Employee employee : employees) {
            totalSalary += employee.getSalary();
        }
        return totalSalary;
    }
}
// Client Code
public class CompositePatternDemo {
    public static void main(String[] args) {
        // Create individual employees (leaves)
        Employee dev1 = new Developer("John", 80000, "Senior Developer");
        Employee dev2 = new Developer("Sarah", 75000, "Developer");
        Employee dev3 = new Developer("Mike", 70000, "Junior Developer");
        Employee dev4 = new Developer("Lisa", 85000, "Senior Developer");
        // Create departments (composites)
        Department engineering = new Department("Engineering");
        Department backendTeam = new Department("Backend Team");
        Department frontendTeam = new Department("Frontend Team");
        // Build hierarchy
        backendTeam.addEmployee(dev1);
        backendTeam.addEmployee(dev2);
        frontendTeam.addEmployee(dev3);
        frontendTeam.addEmployee(dev4);
        engineering.addEmployee(backendTeam);
        engineering.addEmployee(frontendTeam);
        // Treat composite and leaf uniformly
        engineering.showDetails();
        System.out.println("\n\nJust Backend Team:");
        backendTeam.showDetails();
    }
}
C++ Implementation
cpp
#include <iostream>
#include <vector>
#include <string>
#include <algorithm>
// Component
class Employee {
public:
    virtual void showDetails() = 0;
    virtual int getSalary() = 0;
    virtual ~Employee() {}
};
// Leaf
class Developer : public Employee {
private:
    std::string name;
    int salary;
    std::string position;
public:
    Developer(const std::string& name, int salary, const std::string& position)
        : name(name), salary(salary), position(position) {}
    void showDetails() override {
        std::cout << position << ": " << name << " ($" << salary << ")" << std::endl;
    }
    int getSalary() override {
        return salary;
    }
};
// Composite
class Department : public Employee {
private:
    std::string name;
    std::vector<Employee*> employees;
public:
    Department(const std::string& name) : name(name) {}
    void addEmployee(Employee* employee) {
        employees.push_back(employee);
    }
    void removeEmployee(Employee* employee) {
        employees.erase(
            std::remove(employees.begin(), employees.end(), employee),
            employees.end()
        );
    }
    void showDetails() override {
        std::cout << "\n" << name << " Department:" << std::endl;
        std::cout << "------------------------" << std::endl;
        for (Employee* employee : employees) {
            employee->showDetails();
        }
        std::cout << "Total Department Salary: $" << getSalary() << std::endl;
    }
    int getSalary() override {
        int totalSalary = 0;
        for (Employee* employee : employees) {
            totalSalary += employee->getSalary();
        }
        return totalSalary;
    }
    ~Department() {
        // Note: In a real application, you'd need to decide on ownership
        // This example assumes the department doesn't own the employees
    }
};
// Client Code
int main() {
    // Create individual employees (leaves)
    Developer* dev1 = new Developer("John", 80000, "Senior Developer");
    Developer* dev2 = new Developer("Sarah", 75000, "Developer");
    Developer* dev3 = new Developer("Mike", 70000, "Junior Developer");
    Developer* dev4 = new Developer("Lisa", 85000, "Senior Developer");
    // Create departments (composites)
    Department* engineering = new Department("Engineering");
    Department* backendTeam = new Department("Backend Team");
    Department* frontendTeam = new Department("Frontend Team");
    // Build hierarchy
    backendTeam->addEmployee(dev1);
    backendTeam->addEmployee(dev2);
    frontendTeam->addEmployee(dev3);
    frontendTeam->addEmployee(dev4);
    engineering->addEmployee(backendTeam);
    engineering->addEmployee(frontendTeam);
    // Treat composite and leaf uniformly
    engineering->showDetails();
    std::cout << "\n\nJust Backend Team:" << std::endl;
    backendTeam->showDetails();
    // Cleanup
    delete dev1;
    delete dev2;
    delete dev3;
    delete dev4;
    delete backendTeam;
    delete frontendTeam;
    delete engineering;
    return 0;
}
When to Use the Composite Pattern
✅ Use when:
You need to represent part-whole hierarchies
You want clients to ignore the difference between compositions and individual objects
You have tree-like structures (files/folders, UI components, organization charts)
❌ Avoid when:
Your hierarchy is flat (no nesting needed)
Leaf and composite objects have very different interfaces
You need type-specific behavior that can't be generalized
Key Benefits
Simplifies client code: Treat all objects uniformly
Easy to add new components: Just implement the component interface
Natural tree traversal: Operations propagate automatically through the hierarchy
The pattern's power lies in its simplicity: one interface, recursive composition, uniform treatment.
